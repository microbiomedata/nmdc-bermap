Settled shape — catalog-grain resource registry

RB-TnSeq · AMP2/M2P2 · Phage Foundry, modelled at the KB's actual grain.

This knowledge base operates at program / paper / catalog grain, not at bench grain. Once we accepted that, the modelling problem collapsed: studies stay paper-shaped (existing studies / NMDCStudyReference machinery), umbrella-study programs like AMP2/M2P2 reuse nmdc_umbrella_study with arbitrary-bolus datasets, and the only genuinely missing surface was a place to enumerate the large-grain, catalog-level resources a program maintains or produces (mutant libraries, phage banks, knowledgebases) with typed external identifiers. The additions below are the entire diff.

AMP2 / M2P2 · EMSL phenotyping production facility

No new schema needed for the program structure — data releases reuse nmdc_umbrella_study + studies/datasets at whatever bolus size EMSL releases. A single ResearchResource entry catalogs the facility-level capability with an EMSL project ID.

# Under EMSL (UserFacility):
research_resources:
- name: M2P2 — EMSL Microbiome Molecular Phenotyping Platform
  description: >-
    EMSL production phenotyping facility offering integrated multi-omics
    characterization of microbial isolates and consortia submitted by
    user-projects.
  resource_type: KNOWLEDGEBASE
  provider: EMSL (Pacific Northwest National Laboratory)
  url: https://www.emsl.pnnl.gov
  identifiers:
  - registry: internal
    accession: emsl-m2p2
    url: https://www.emsl.pnnl.gov
  # actual data releases go in calling programs' studies/datasets under
  # nmdc_umbrella_study, not here.

Out of scope (intentionally). The earlier sketches explored an AssayPlatform / ExperimentalCondition / Perturbation lattice that would have modeled RB-TnSeq at bench grain (library × condition × Time0 → fitness matrix). That was the wrong grain for this KB. The fine-grained machinery is dropped; if a future use case needs assay-level decomposition, it should be a separate, downstream schema and not a generalization of this one.
